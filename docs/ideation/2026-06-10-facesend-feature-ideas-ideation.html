Grounding Context

Codebase context. FaceSend is a fully client-side photo distribution app: Next.js 15 + TypeScript, @vladmandic/face-api running in-browser (WebGL), IndexedDB via idb, client-zip for downloads. Flow: upload ~300 photos → sequential per-photo detection (~1s each, two-pass: 640px fast detect + 400px high-res chip descriptors) → greedy centroid clustering at threshold 0.45 (deliberately stricter than the canonical 0.6 same-person threshold, because the UI can merge but not split) → review cards (4 face chips, name + phone/email free-text, Skip, multi-select merge) → per-person /p/[personId] links. v1 links are local-only — they resolve from the organizer's IndexedDB, so recipients on other devices hit a dead end. No SMS/email sending, no undo; replaceClusters() rewrites all cluster assignments from scratch, discarding manual merges. A tap-to-eject split feature is being specified separately. Photos with zero detected faces reach no one.

External context. Competitors (Waldo, Premagic, Kwikpic, Pic-Time, Memzo, Fotify) lead with selfie-matching guest portals, QR codes at the venue, WhatsApp/SMS delivery, download locks, and sponsor watermarks — all server-side. None markets on-device processing; FaceSend's architecture is genuine white-space. Cross-domain signals: in mail-merge-shaped products the contact list is the bottleneck and the durable asset; stadium print kiosks showed delivery latency matters emotionally; Firefox Send/Excalidraw established the encrypted-blob + key-in-URL-fragment pattern for zero-trust hosting.

1

Doubt Queue — confidence-ranked correction

accuracy & correctionConfidence 90%Complexity Medium
Description
Make review time scale with model uncertainty, not photo count. After clustering, surface only what the algorithm is unsure about: cluster pairs whose centroid distance falls in the 0.45–0.60 band as one-tap "Same person?" side-by-side cards, plus a second-pass rescue sweep matching orphan singletons (and faces previously dropped for being under 48px) against the now-labeled centroids — "We found 12 more photos that look like Priya — accept / reject." The confident 95% is never shown.
Basis
direct: src/lib/face/cluster.ts documents threshold 0.45 as "deliberately stricter than the canonical 0.6 same-person threshold" because under-merging is the recoverable failure mode — the system knowingly manufactures duplicates in a known distance band, and the pairwise distances needed to rank suggestions are already computed in clustering pass 2.
Rationale
The codebase already encodes exactly where mistakes concentrate, then throws that information away and makes the host find duplicates by eyeballing a wall of cards. This converts an O(people²) visual hunt into a 30-second yes/no queue — and it complements the smart-eject split feature being specified now (eject fixes over-merges; the queue fixes under-merges).
Downsides
Band tuning needed to keep suggestions useful rather than noisy; adds a step before the review grid that must be skippable.
2

Corrections Ledger — fixes that survive re-clustering

accuracy & correctionConfidence 85%Complexity Medium
Description
Record every host correction — a merge, a smart-eject, a Doubt Queue answer — as a persistent must-link / cannot-link constraint stored beside the descriptors. Any future clustering run enforces the constraints (constrained greedy clustering, COP-KMEANS-style), so re-clustering can never resurrect a mistake the host already fixed.
Basis
direct: replaceClusters() in src/lib/db.ts clears the entire clusters store and reassigns every face from scratch — any manual merge is silently destroyed by the next clustering pass. Corrections currently live only as disposable UI state.
Rationale
A human judgment is the most expensive signal in the app, and today it has the shortest lifespan. The ledger is the compounding primitive every other accuracy feature sits on — and it's the hard prerequisite for "add more photos later" and for Photo Potluck (idea 8), both of which require safe re-clustering.
Downsides
Constraint-aware clustering is subtler than the current pure function; conflicting constraints need a resolution rule.
3

Guest Self-Claim — selfie labeling that never leaves the device

labeling & contactsConfidence 80%Complexity Medium-High
Description
Invert who does identification. A "Find me" kiosk mode on the host's device (or pass-the-phone): a guest taps, takes a selfie, the descriptor is computed locally and matched against cluster centroids — the same euclideanDistance path that already exists — and the guest types their own name and contact into their matched cluster. The host labels only the stragglers.
Basis
external: Waldo, Kwikpic, and Premagic all ship selfie-matching portals as their headline guest UX — but every one uploads the selfie to their servers. FaceSend can deliver the identical interaction with the selfie never leaving the device it was taken on.
Rationale
Labeling is the worst-scaling step (the host must know and correctly spell ~N names and contacts, and at any event bigger than a dinner party they provably don't know everyone). Self-claim doesn't speed the bottleneck up — it deletes it, and the guest types better data about themselves than the host ever could. It turns the biggest manual cost into the biggest privacy differentiator in one feature.
Downsides
Adds a camera surface and a guest-facing locked mode to a host-only app; one-shot selfies in bad lighting may need a retry flow.
4

Roster Import + Contact Picker autofill

labeling & contactsConfidence 90%Complexity Low-Medium
Description
Let the host paste or upload the guest list they almost certainly already have (RSVP export, CSV, spreadsheet paste). Labeling becomes a type-ahead pick — contact auto-fills. On Android Chrome, add navigator.contacts.select() so a card is labeled from the host's real address book in one tap. Unmatched roster entries become a report no competitor offers: "no photos found for these 12 guests."
Basis
direct: src/lib/contacts.ts validates only free-typed name + phone/email today; the mail-merge research frames the contact list as the bottleneck. external: W3C Contact Picker API, shipped in Chrome/Android since v80, returns exactly the {name, tel, email} shape ContactInfo already stores.
Rationale
Typing 30 names and 30 phone numbers on a phone keyboard is the most error-prone, tedious moment in the flow, and the data already exists in the host's spreadsheet or contacts app. Cheapest large win in the set, and it seeds the People Directory (idea 5) for free.
Downsides
Contact Picker is Android-Chrome-only (iOS needs the CSV/paste path); CSV shape diversity needs forgiving parsing.
5

People Directory — label once, recognize forever

labeling & contactsConfidence 70%Complexity Medium
Description
A durable people store — name, contact, and a small set of representative descriptors per person — kept separate from per-event clusters and exempt from event reset. At the next event, fresh clusters are matched against the directory and arrive pre-labeled; the host confirms instead of typing. Each confirmed event feeds descriptors back, making future matches stronger. Per-person delete keeps it trustworthy.
Basis
direct: descriptors are already persisted per face precisely so re-clustering never re-runs inference, and contacts are already stored per cluster — the asset exists and is currently destroyed between events by resetAll().
Rationale
For the highest-value users — repeat hosts: clubs, leagues, schools, photographers — labeling cost drops toward zero after event one. It converts a one-off tool into a system with switching costs, and no cloud competitor can copy it while claiming the directory never left your device.
Downsides
Persistent biometrics raise the privacy stakes — needs explicit opt-in framing and pairs naturally with the retention controls in idea 10; browser storage eviction risk argues for pairing with an export/backup path.
6

Portable Gallery Bundles — the link becomes a file

delivery + arrivalConfidence 90%Complexity Low-Medium
Description
Generate each person's deliverable as a self-contained artifact: their photos plus an embedded index.html gallery (named header, grid, lightbox) that reproduces the share-page experience offline on any device. "Send" becomes the OS share sheet — AirDrop, WhatsApp, email attachment — via the Web Share API, which on mobile also offers save-to-camera-roll instead of a zip lost in the Files app.
Basis
direct: the share page's not-found state literally instructs recipients to "open this link on the device that created it" — the app currently tells users to work around its own delivery mechanism — and client-zip already assembles per-person zips, so the bundle is an incremental change to an existing path.
Rationale
v1's local-only links mean FaceSend technically cannot deliver photos to anyone else's device — the core promise of a distribution app. This is the smallest-diff fix to the biggest gap, it requires no server (deepening the on-device story), and the recipient gets a gallery they own outright: no hosting, no expiry, no account.
Downsides
Large events produce large files (sharing-channel size limits apply); single-file HTML with base64 photos has a practical size ceiling, so zip + index.html is the safer default shape.
7

Zero-Knowledge Photo Locker — real links, no trusted server

delivery & sendingConfidence 70%Complexity High
Description
Encrypt each person's bundle client-side (AES-GCM), upload the opaque blobs to any dumb static host (S3, Netlify, even Drive), and put the per-person decryption key in the URL #fragment — which browsers never send to servers. Recipients get normal-feeling links that work anywhere, while neither FaceSend nor the storage provider can see a single photo or attribute one to a person.
Basis
external: the established end-to-end pattern of Firefox Send, Wormhole.app, and Excalidraw share links: ciphertext on a dumb server, key in the fragment, decryption purely client-side.
Rationale
This resolves the product's core tension — links are local-only because there's no server, but a normal server would kill the privacy identity. Fragment-key encryption is the one path to remote links that keeps the zero-trust story fully intact, and it composes with bundles (6) and self-claim (3) into a complete delivery suite. A P2P WebRTC variant (host-online-only) was considered and is noted in the rejection table as the synchronous alternative.
Downsides
Someone still has to host blobs (host-provided storage or a paid tier — a real product decision); upload bandwidth from the host's device; key-in-URL means anyone forwarded the link gets access, so it's privacy-from-infrastructure, not access control — pair with selfie verify (3) if gating matters.
8

Photo Potluck — recipients bring photos back

arrival experienceConfidence 60%Complexity High
Description
After a recipient views their photos, the share surface invites a return gift: contribute their own phone photos of the event back into the pool. Contributions get face-processed and redistributed to everyone who appears in them — the share page stops being a receipt and becomes a two-way exchange.
Basis
reasoned: the host's camera captured one vantage point while every guest's phone captured others; the people arriving at the share page are, by definition, exactly the population holding the missing photos — at the precise moment they're most engaged with the event. That is the cheapest possible acquisition of new supply.
Rationale
The only idea in the set that makes the product better the more it's used: each arrival can grow the pool, which triggers more deliveries, which triggers more arrivals. It is also FaceSend's only organic growth loop, since there's no server-side viral channel.
Downsides
Requires a working return channel (bundles/locker first) and safe append-and-recluster — which is why the Corrections Ledger (2) is a prerequisite; host moderation of contributions is needed.
9

Consent Shield — per-recipient bystander redaction

privacy & trustConfidence 65%Complexity Medium
Description
Let the host mark any person "do not distribute." At delivery time, that person's faces are blurred in every other recipient's copies via a canvas pass — each recipient gets a personalized, consent-respecting version of the same group photo. Skipped and never-labeled clusters can optionally get the same treatment.
Basis
direct: FaceRecord.box stores per-face bounding boxes in original-image pixel coordinates and ClusterRecord.skipped already exists — every input needed for targeted blur is in IndexedDB today.
Rationale
Group photos leak third parties by construction: every photo sent to person A also hands them persons B–F, who never consented. Because rendering already happens per-recipient on-device, per-copy redaction is nearly free here while being cost-prohibitive for server pipelines — the first feature competitors structurally can't match cheaply. For schools, kids' events, and GDPR-sensitive hosts, this is the purchase trigger.
Downsides
Re-encoding photos per recipient costs CPU and some image quality; needs a sensible default (off, per-event toggle) and a feasibility spike on throughput.
10

Airplane-Mode Proof — make the privacy claim verifiable

privacy & trustConfidence 85%Complexity Low-Medium
Description
Turn the architecture into a demonstrable feature: a service worker pre-caches the app and the ~12MB of models, and the UI actively invites the host to flip on airplane mode before processing — and keeps working. Add a live network meter (every outbound request itemized; should be only model fetches), and close each event with a privacy receipt: N photos, M faces, processed on this device, zero bytes uploaded — plus an optional retention sunset that auto-purges unclaimed face data after a host-set window, with the expiry date visible to recipients.
Basis
direct: the market research found that none of the six researched competitors markets on-device processing — identified white-space — yet nothing in FaceSend's UI surfaces the property at all today. The claim exists only in the README.
Rationale
A differentiator the user can't perceive doesn't differentiate. "Turn off your Wi-Fi and watch it still work" is the rare privacy claim a non-technical host can verify in five seconds, and no cloud competitor can ever copy the demo. It converts an architectural decision into the marketing headline and gives every other idea in this set its trust foundation.
Downsides
Service-worker caching adds update-flow complexity (stale-app traps); the network meter must be honest about model downloads to avoid undermining the claim it exists to make.
